# deque
## 底层核心
1. 中控器(Map)
   - 它是一个连续的指针数组(底层类型通常是T) 分配在堆区，数组里的每一个元素，都是一个指向真正存放数据的“缓冲区”的指针
2. 缓冲区(Buffer/Node)
   - 本质：这是一段段固定大小的连续堆内存(在常见的实现中，比如GCC，每一个buffer通常是512字节)
   - 所有的实际数据都按顺序存放在这些分散的缓冲区里
3. 布局
[栈区 Stack]
deque 对象本身 (包含指向中控器的指针等状态信息)
      |
      V
[堆区 Heap]
================= 中控器 (连续指针数组) =================
        [ ptr0 ]    [ ptr1 ]    [ ptr2 ]    [ ptr3 ]
           |           |           |           |
           |           |           |           V
           |           |           |      [ 缓冲区 3 ] (固定大小连续内存)
           |           |           V      [ _ _ _ _ ] <- 可以在尾部继续 push_back
           |           |      [ 缓冲区 2 ]
           |           V      [ X X X X ] <- 存满了真实数据
           |      [ 缓冲区 1 ]
           V      [ X X X X ] <- 存满了真实数据
      [ 缓冲区 0 ]
      [ _ _ X X ] <- 可以在头部继续 push_front

## 设计平衡
对比vector头插的O(n)算法 deque可实现为O(1)
即直接在缓冲区0放入 即便是满了 也只需要再向操作系统申请一小段新的Buffer，然后把新的Buffer的指针记录在中控器map的更前面一个位置即可。全程没有任何旧数据的拷贝与搬迁

## 底层指针
```cpp
template <class T, ...>
struct __deque_iterator {
    T* cur;             // 指向当前 Buffer 中的当前元素
    T* first;           // 指向当前 Buffer 的起始位置
    T* last;            // 指向当前 Buffer 的结束位置（前闭后开）
    map_pointer node;   // 这是一个二级指针！指向中控器 map 里的那个格子
};
```
在执行++it时 会经历一系列操作
1. ++cur
2. 判断if(cur == last)
    1. 如果不是 退出
    2. 如果是 则需要将角色(cur)传到下一个关卡(node)
    3. 放到起点cur = first
由于这些判断 就让deque的遍历速度永远追不上vector

## operator[]
> 由于deque头部不一定在0的位置 所以需要引入一个起始偏移量offset
1. 调用dq[200]时 计算公式是(假设offset=30)
   1. 绝对位移：absolute_pos = offset + 200 = 30 + 200 = 230
   2. 跨越的Buffer数量：230 / 128 = 1 (即在中控器上 距离start.node要往后走一格)
   3. 在目标Buffer内的具体槽位: 230 % 128 = 102

## 头尾操作
1. push_back()的底层逻辑
   - 迭代器去检查最后一个Buffer，如果还有空位，直接构造对象
   - 如果最后一个Buffer已经写满了，向空间配置器申请一段新的纯洁Buffer，把Buffer的地址记录到中控器map的下一个格子里，然后把数据写进新的Buffer的开头
2. push_front()
   - 迭代器检查头部 如果有空位，直接写入
   - 如果头部已经到边界，申请一块新的Buffer，将地址记录到map的前一个格子里，把数据写在新Buffer的末尾

## 扩容机制
1. 缓冲区(Buffer)增生
    - 数据满了 就多建一个BUffer，原来所有的数据都在各自的Buffer里原地不动
2. 中控器(Map)扩容
    - 当中控器也满了时，申请一块更大的指针数组(比如容量翻倍)，然后把旧map里的指针拷贝到新map的中间位置(为了给头尾不都预留扩展空间)，最后释放旧的指针数组

## 迭代器失效问题
### 1.插入操作
1. 头尾插(push_back,push_front): 所有迭代器失效(因为中控器map可能扩容导致node野指针)，但所有原生指针和引用绝对有效(因为数据所在的Buffer没动)
2. 在中间插入(insert): 由于发生了数据平移，所有的迭代器，所有的原生指针 所有的引用 全部失效
### 2.删除操作
1. 在头尾删除(pop_back,pop_front)：极度安全。只有被删除的那个元素迭代器失效 其他所有都有效
2. 在中间删除(erase)：同样因为触发了数据跨Buffer平移，所有迭代器，所有原生指针引用全部失效



## 中间删除的坏处
1. 跨越缓冲区的艰难平移
    - 在vector中，如果在中间插入一个元素，底层只需要依次高效的内存拷贝函数(如底层c语言的memmove)，把后面的元素整体往后推一格，因为它们在物理内存上是绝对连在一起的
    - 而deque里 数据被切碎在了一个个512字节的Buffer里，插入一个数据时
        1. 需要判断是从前面挤出空间，还是从后面挤出空间(deque会聪明地比较插入点离头近还是离尾近，挑选挪动数据较小的一端进行平移)
        2. 在平移的过程中 数据不能简单地整块拷贝 而是**跨越Buffer边界** 当一个元素被挤出当前Buffer时，他必须被精准地投递到下一个或上一个Buffer的对应位置
        3. 涉及海量的迭代器指针边界判定与跳跃计算
