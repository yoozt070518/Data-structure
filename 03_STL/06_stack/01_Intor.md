# stack
## 1. 宿主选择
对于默认的stack是以deque作为底层去实现的
但我们也可以自己选择宿主
- std::stack<int, std::vector<int>> s_vec; // 强行用 vector 当底座
- std::stack<int, std::list<int>> s_list;  // 强行用 list 当底座

### 为什么选择deque
deque的物理结构：中控器Map + 离散缓冲区Buffer
- **完美契合栈的操作**：栈只需要在尾部（一端）疯狂进行进出操作。deque的尾部插入和删除是绝对的O(1)
- **无敌的扩容机制**：当尾部的Buffer满了，deque只需要向操作系统申请一块小块的新的Buffer,然后把指针挂在中控器上。它绝对不会发生全量数据的内存搬迁
- **内存利用率极高**：不需要像list那样每个元素额外携带

### 强行用vector的坏处
如果写了std::stack<int, std::vector<int>> s;，平时用着感觉极快，因为内存绝对连续。但它有一个致命的缺陷：**扩容灾难**
后果：在工业界（如高频交易，游戏服务器），这种偶尔爆发一次的O(n)级别全量搬迁，会导致系统出现不可预知的延迟毛刺。前9999次push只花了1纳秒。对于栈这种要求极致稳定O(1)的结构，这是不可接受的

### 强行用list的坏处
如果你写了 std::stack<int, std::list<int>> s;确实解决了vector的扩容搬迁问题。list的每次push_back都是绝对的常熟时间。但它引来了两个更致命的性能杀手
- 疯狂的new和delete：list是绝对离散的。这意味着每一次压栈（push）,底层都要向操作系统呼叫一次new分配内存，每一次弹栈(pop)，都要呼叫一次delete。频繁的系统调用极其极其缓慢，相比之下，deque是按块(Buffer)分配的，几百次压栈才需要向操作系统申请一次内存
- 极低的内存有效负载率：如果是来存int的，list为了存放这个int还要加上prev和next指针，加上内存对齐，一个节点硬生生到了24字节，空间极度浪费
- Cache Miss: 虽然栈主要访问栈顶元素，不需要遍历，但list碎片化的内存依然会让CPU缓存很不舒服

### 为什么使用deque的回答：
“因为 deque 完美融合了 vector 和 list 的优点。它避免了 vector 扩容时全量搬迁的延迟灾难，同时它是按块分配内存的，又避免了 list 频繁 new/delete 单个节点带来的系统调用开销和内存碎片问题。”

## API调用
stack<int> s;
- s.push(val);
    - 压栈，把元素放到栈顶 底层调用deque::push_back.代价：O(1)
- s.pop();
    - 弹栈。把栈顶元素销毁。底层调用deque::pop_back。代价O(1)
    - 注意：**它没有返回值**
- s.top();
    - 查看栈顶。获取栈顶元素的引用。底层调用deque::back。代价O(1)
- s.size();
    - 获取元素个数
- s.empty()
    - 判空

### 为什么要把pop和top拆开
在Java python等语言中pop()函数通常会：**不仅删除栈顶元素，还同时把这个元素作为返回值给出来**
- 原因：为了极其严苛的异常安全
  - 在返回值的时候 如果这个返回的变量是一个很大的结构体 程序需要把值拷贝给调用者。如果在这个拷贝的瞬间，拷贝构造函数抛出了一个异常
    1. 拷贝失败，调用者没有拿到这个对象
    2. 但是，第二步c.pop_back()已经执行过了，栈里的原件已经被彻底销毁了
    3. 结果：这个数据被永久丢失了
