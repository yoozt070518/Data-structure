# list
- 底层结构：带头结点的双向循环链表
## 核心结构
- list是一个绝对的碎片化与分散
```cpp
template <class T>
struct _List_node {
    _List_node<T>* prev; // 指向前一个节点 (8字节，64位系统下)
    _List_node<T>* next; // 指向后一个节点 (8字节)
    T data;              // 真实存放的数据
};
```
1. 每一个list都存在一个虚拟头节点 Dummy Node 它的next就是第一个节点 它的prev就是最后一个节点 
2. begin() -> 第一个节点  end() -> 虚拟头节点

## 内存对齐机制
- CPU在读内存的时候不是一个字节一个字节去读的 而是按块来读的 在64位系统下，CPU一次通常读取8个字节
1. 原因：最长的基础类型double占了8个字节 如果一次只读4个字节的话double的前半段和后半段在不同的读取块中
2. 空间换时间：为了让CPU每次都能舒服的一次性读出完整数据，c++编译器会在后台偷偷做手脚：向结构体里塞入毫无意义的“垃圾数据(Padding)”来占位
3. 对齐的基本规则：又叫最宽数据类型原则
   - 对于包含不同数据类型的结构体，编译器通常会寻找占用字节数最大的哪一个基础成员(在节点中就是8个字节的指针)
   - 然后，编译器会强行要求：整个结构体的总大小，必须是这个最大成员(8字节)的整数倍

## list的内存极度浪费
- 对于储存数据的节点，每个节点的数据占用的字节数为8+8+4 = 20，根据最宽数据类型原则 一个节点占用了24个节点，而一个int只需要4个字节 这样就是内存的利用率极其低下 只有4/24 = 1/6

## 阶级分明的迭代器
1. 在vector和deque中，它们的迭代器属于最高级的随机访问迭代器(Random Access Iterator)，只要写it+5 底层就会直接调用数学加法(配合offset)瞬间算出内存地址。但在list中，物理内存是碎片化的，它的迭代器退化成了双向迭代器(Bidirectional Iterator)
2. 它支持：++it，--it
3. 禁止：it+5，it+=3,it1-it2
4. 性能代价：list的遍历和查找 永远都是铁打的O(N);

## 指针与迭代器安全性
- 对于vector和deque都会因为insert等操作出现指针和迭代器失效问题
- 但这在list这里变得不复存在
1. 任何的插入操作(insert,push_back,push_front):
   仅仅是new出来的一个新的_List_node节点，然后去拉扯修改附近的两个节点的prev和next指针
   - 结论：所有迭代器，原生指针，引用，绝不会失效
2. 任何删除操作(erase, pop_back,pop_front):
    仅仅是把删除的那个节点从链表上解下，然后delete掉
    - 结论：除了被删除的那个节点的迭代器/指针会变成野指针失效外，其他所有的迭代器，指针，引用都绝对有效
- 因此，在某些需要长期持久化保存特定元素指针的业务系统中，list能有一席之地

## splice
- 在vector中，如果向把一个包含10w个元素的vector A的一般数据，移动到vector B的头部，这是一场灾难，时间复杂度到了O(n)
- 但是在list中，只需要把B中间某段节点的前后prev/next指针剪断，然后A的指定位置上，全程只修改4到6个指针的值，无论是移动一个还是100w个元素，时间复杂度永远是O(1)
### 核心用法
1. 把整个链表都搬走 
    - void splice(const_iterator pos, list& other);
      - pos:目标链表的一个迭代器，它决定了拼接到哪里，数据会精准地插入到这个迭代器所指元素地前面
      - other:要掏空的链表
2. 只搬运链表里的“单个节点”
    - void splice(const_iterator pos, list& other, const_iterator it);
      - pos:目标位置（前面）
      - other：源链表对象
      - it:other的一个有效迭代器 代表要精准切下哪一个节点
3. 搬运源链表里的一段区间
    - void splice(const_iterator pos, list& other, const_iterator first, const_iterator last);
    - 左闭右开区间
注：版本2，3可以用来对链表进行自身移动

## sort函数
- list由于独特的双向迭代器，不能使用像vector那样的随机访问迭代器，也就不能做到sort快排里的随机访问能力
- list.sort();
  - 底层算法：它底层并没有使用快排，而是使用了一种极其适合链表特性的算法-->非递归的归并排序
  - 在排序的过程中 它依然不会发生任何数据的拷贝和移动，它纯粹是在修改指针的指向，把节点重新“串”成有序的形状。时间复杂度依旧是O(nlogn)

## CPU Cache命中率极其低下
- 在大学课堂中，有这样一句话“如果业务场景需要频繁在中间进行插入和删除，请使用链表（list），因为它是 O(1)；绝对不要用数组（vector），因为它是 O(N)。”
- 但是在实际开发中 即便是中间频繁增删，我们还是会选择vector
原因：
1. **CPU缓存与主存**
- CPU运行速度是极快的，但内存(RAM)相对来说慢得像蜗牛(上百纳秒)，为了让CPU不那么闲着等内存传数据，现代CPU内部封装了一层极其昂贵但极速的硬件缓存(L1/L2/L3 Cache)
- CPU读数据的逻辑：
    1. 先去Cache里找，找到了叫命中(Cache Hit)，瞬间拿到数据，CPu狂飙
    2. 找不到叫未命中(Cache Miss),CPU必须强制停机等待，去等主存拉数据，这会浪费成百上千时钟周期
2. 空间局部性(Spatial Locality) 与 Cache Line 机制
    **这是vector碾压list的根本原因**
    - 当CPU发生Cache Miss去读内存时，它绝对不会只读你想要的那一个字节。为了效率，它会把那个字节相邻的一大块连续内存(通常是**64字节**，称为一个Cache Line)全部打包塞进Cache里
    - 而vector因其连续的物理特性，使CPU的Cache Hit命中率极高 相反的，list因其分散的物理结构，让CPU的Cache Hit命中率极低 从而导致了list比vector寻找位置加上平移的总时间要长上几倍或者几十倍

针对巨型结构体（如 Boss），大厂真正的解法是使用 vector<指针> (如 vector<Boss*>)
