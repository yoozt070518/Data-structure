# 哈希表
## 1. 为什么要造出哈希表
- 动态数组：只给定索引(如arr[5]),CPU就能通过基地址+索引*元素大小的指针偏移，在O(1)的时间复杂度内直接拿到内存中的数据。这种极速的随机访问是数组的杀手锏
- 然而，在实际工程中，我们查找数据往往不是通过"索引"，而是通过具体的内容或键
    - 比如通过学号查找学生对象，或者通过字符串"ALice"查找到他的考试分数
如果用现有的数据结构：
- 数组/链表：必须从头遍历对比，时间复杂度退化成O(n)。
- 排序数组+二分查找：虽然能降到O(logn)，但插入和删除数据时需要挪动大量内存空间，依然非常耗时

哈希表的本质：它底层依然是一个连续的内存数组。它的野心在于，既要保留数组O(1)的随机访问速度，又要打破"只能用数字索引"的限制。它要实现输入一个任意类型的Key(比如字符串"Alice"),瞬间能知道这个Key应该存放在底层数组的哪个物理位置

## 2.哈希函数：连接Key与内存的桥梁
1. 哈希计算(Hashing)：将传入的任意类型的Key(如字符串，复杂对象)经过特定的算法，转化成一个巨大的整数，这被称为哈希值(Hash Code)
2. 取模映射(Modulo Mapping): 算出的哈希值可能非常大(比如2^31-1).我们不可能在内存里开辟这么大的底层数组。因此，必须用哈希值对当前底层数组的容量进行取模运算，得到一个合法的数组索引Index

Index = Hash(Key) % Array_Capacity
- 底层视角的优秀哈希函数三大标准：
  1. **确定性**：对于相同的Key不管计算多少次，输出的哈希值必须绝对一致。否则存进去的数据永远也找不回来
  2. **高效性**：计算过程要极度轻量。
  3. **均匀性(最核心)**：尽可能把不同的Key打散，让它们经过计算后均匀地落到底层数组地各个角落，避免扎堆

## 哈希冲突解决策略
### 1.开放寻址法：数组内的抢座游戏
核心逻辑：绝不借助外部数据结构。如果当前算出来的Index已经被别人占了，那就按照某种规则，在当前的底层数组中继续寻找下一个空闲地坑位
- 线性探测(Linear Probing):
    - 规则：如果Index被占了，就看Index+1，如果还被占，就看Index+2，以此类推，直到找到空位
    - 致命弱点————群聚现象：假设连续几个坑位都被占了，那么后续哈希到这附近地数据都会堆积在一起，形成一个越来越大的"连体婴儿"。这会导致查找时间迅速退化
- 二次探测：
    - 规则：为了解决线性探测的”拥挤“问题，探测步长变为平方数：Index+1^2,Index+2^2...这样就可以快速跳离拥堵区域
开放寻址法的最大优点时极其契合Cpu Cache。因为它所有的数据都实打实地存在一个连续地内存数组中，现代cpu在读取内存时，会把相邻的内存块预取到高速缓存中，所以在线性探测时，CPU命中缓存率极高，时间常数很小

### 2.链地址法：数组与链表的缝合怪
这是工程中最主流的做法，也就是手写哈希表时要采用的方案
- 核心逻辑：哈希表的底层数组不再直接存储数据(Value),而是存储一个指针(通常是指向单向链表头节点的指针)。如果发生了冲突，就把新来的数据作为新节点，挂接到当前Index对应的链表上
    - 插入与查找：定位到数组Index是O(1) 如果在该位置发现链表，则需要遍历这条链表去对比具体的Key

但这种方法不利于CPU Cache

## 3.动态扩容与内存管理
1. 负载因子
   - 为了知道哈希表什么时候"快撑不住了" 底层引入了一个监控指标--负载因子
       - 计算公式：负载因子 = 当前哈希表中存储的总元素个数 / 底层数组容量
       - 物理意义：它反映了底层数组的拥挤程度。比如容量是10 存了七个元素，负载因子就是0.7， 当负载因子越来越大，发生哈希冲突的概率呈指数级上升 
       - 在c++的std::unordered_map中，默认最大负载因子阈值通常是1.0.一旦超过这个警戒线，底层就会强行介入
2. 核心干预：重哈希与动态扩容
当负载因子超标，或者某条链表实在太长时，底层会执行一次极其”昂贵“但必不可少的内存手术：
        - 内存扩容：在堆区申请一块全新的，更大的连续内存数组(通常是原容量的两倍，或者是下一个更大的质数，以保证后续取模的均匀性)
        - 重新映射(重哈希)
